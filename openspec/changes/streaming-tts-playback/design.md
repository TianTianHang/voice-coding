## Context

当前后端已经有较完整的 STT/VAD 架构：录音通过 `cpal` 输入流进入 VAD 状态机，转写完成后再继续保持监听。现在需要新增 TTS 能力，但目标不是“边说边播”，而是“先合成完整音频，再播放”，并且播放期间必须暂停录音，避免扬声器输出被麦克风重新采集后引起回灌和误触发。

本次变化会同时影响音频输入、音频输出、后端运行时状态和 Tauri 命令，因此更适合先明确抽象边界，再进入实现。

## Goals / Non-Goals

**Goals:**
- 建立与 STT 类似的 TTS 抽象层和实现层分离结构。
- 支持文本到音频的合成，并在合成完成后再播放。
- 统一最终播放格式为 48kHz 立体声 PCM。
- 播放期间暂停录音/VAD，播放结束或中断后恢复。
- 让 TTS 状态可查询、可上报、可测试。

**Non-Goals:**
- 不在第一版实现真正的边合成边播放。
- 不设计复杂的回声消除或 barge-in 机制。
- 不把 TTS 播放逻辑直接耦合进 TTS 模型实现。
- 不修改前端交互流程，只提供后端能力与状态接口。

## Decisions

### 1. 将 TTS 抽象单独放入 `tts-core`

选择新建独立核心 crate，而不是复用 `stt-core`。原因是 TTS 的输入/输出语义与 STT 差异明显：STT 面向音频输入和文本输出，TTS 面向文本输入和音频输出。拆分后可以避免类型语义混杂，也方便以后并行支持多个 TTS 提供方。

**备选方案：** 复用 `stt-core` 扩展一套 TTS trait。该方案短期改动更少，但会让核心抽象过于拥挤，后续维护成本更高。

### 2. 合成与播放分离

TTS 引擎只负责合成，播放由独立的音频输出模块负责。引擎可在内部流式地产出 token 或音频块，但对外只在合成完成后返回完整音频结果，由播放层统一消费。

**备选方案：** 在引擎内部直接播放。该方案实现更快，但会把模型、缓存、设备控制、生命周期和错误处理全部耦合在一起，不利于测试和替换实现。

### 3. 输出层使用 `cpal` 的输出流

沿用现有项目对音频设备的直接控制方式，在 Rust 后端通过 `cpal` 创建输出流。这样可以和当前录音能力共享同一类平台适配思路，并减少额外依赖。

**备选方案：** 使用 `rodio` 之类的更高层音频播放库。优点是封装更强，但会引入另一套播放抽象，和现有录音链路不一致。

### 4. 统一内部播放格式为 48kHz 立体声 PCM

TTS 的最终播放格式固定为 48kHz、双声道、PCM。这样可以匹配目标 TTS 输出约束，也便于播放层统一处理。若模型输出与目标格式不一致，则在播放边界进行重采样或声道适配。

**备选方案：** 保留模型原始格式直接输出。该方案减少一次转换，但会把设备适配压力暴露到各个调用点，不利于统一控制。

### 5. 播放期间让录音会话进入暂停语义

播放开始时，后端暂停当前录音/VAD，会话恢复则在播放结束或失败后统一执行。第一版优先保证“绝不回灌”，而不是追求播放与录音并行。

**备选方案：** 仅在 VAD 层忽略播报声音，不暂停录音。该方案对用户体验更复杂，也更依赖回声控制，风险更高。

### 6. TTS 状态机保持简洁

状态采用 `idle -> synthesizing -> ready -> playing -> idle`，失败时进入 `failed`。保留必要的阶段信息即可，不提前引入流式播放、排队、多任务并发等复杂状态。

**备选方案：** 加入 buffering / draining / queued 等更多状态。该方案对调试更细，但第一版过度设计。

## Risks / Trade-offs

- [设备格式不支持 48kHz 立体声] → 在输出层做格式适配，并在状态/错误中保留设备信息。
- [合成耗时较长导致播放延迟] → 允许合成阶段上报进度，后续可再考虑更细粒度缓存，但第一版不做边播边合成。
- [播放状态切换时录音恢复失败] → 把录音暂停/恢复设计为幂等操作，并在失败路径上统一清理会话状态。
- [音频回灌导致误触发] → 第一版采用播放期间彻底停止录音的策略，优先保证稳定性。
- [新增模块导致测试面扩大] → 通过状态机单测、格式转换测试和播放生命周期测试覆盖关键路径。

## Migration Plan

1. 新增 `tts-core` 与 TTS 实现 crate，先完成抽象和基础合成接口。
2. 新增音频输出模块，验证 48kHz 立体声 PCM 播放链路可用。
3. 新增 TTS runtime 与 Tauri 命令，串起合成、播放、暂停录音、恢复录音。
4. 将播放期间录音暂停接入现有 VAD/录音会话控制。
5. 补齐 Rust 单测与集成测试，再逐步接入前端调用。

**Rollback strategy:**
若播放链路不稳定，可先保留 TTS 合成能力，暂时关闭播放命令和录音暂停联动，仅保留状态查询与合成结果输出，避免影响现有 STT/VAD 主流程。

## Open Questions

- TTS 实现层是采用本地模型还是远端服务？这会影响 crate 边界和错误处理方式。
- 合成结果是优先返回内存 PCM，还是也要同时支持临时文件导出？
- 录音暂停时，前端是否需要展示明确的“播报中”状态，还是只依赖后端事件即可？
- 未来是否需要支持多个音色/说话人？如果需要，`TtsConfig` 需要提前预留哪些字段？
