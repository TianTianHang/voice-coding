## Context

当前 `assistant-console-ui` 已覆盖会话状态、事件流、确认交互与托盘行为，但页面的信息组织仍以“并列卡片 + 控制按钮”为中心，语音主路径的在场感与激活反馈不足。用户希望获得“纯语音优先”的交互体验，即默认通过唤醒词与连续语音完成操作，界面仅提供必要的可见反馈与兜底控制。

现有前端已经具备可复用能力：
- VAD 状态来源稳定（idle/listening/recording/processing）。
- Agent 事件流已按类型归类并支持 confirm 等关键交互。
- Tailwind 样式体系已建立，便于统一视觉 token 和状态样式。

因此本次设计重点不是新增底层能力，而是将现有能力重构为“语音存在感优先”的体验层，并建立可测试的状态语义与反馈闭环。

## Goals / Non-Goals

**Goals:**
- 建立语音优先的主界面层级：Presence（在场）> Activation（激活）> Understanding（理解）> Response（回应）> Timeline（历史）。
- 定义统一的体验状态机（Dormant、WakeDetected、Listening、Processing、Responding、Error）并映射到现有 VAD/Agent 状态。
- 提供可感知的“激活仪式”与状态过渡反馈，确保用户能判断助手是否被唤醒、是否正在处理。
- 保留最小可信闭环（Heard/Intent/Status/Response），降低纯语音模式下的黑箱感。
- 将低频设置与兜底控制降级，避免干扰语音主流程。

**Non-Goals:**
- 不在本次变更中引入新的 ASR 引擎、VAD 算法或后端协议。
- 不实现完整可视化波形编辑器或多轨音频分析能力。
- 不重构 ACP runtime 协议结构，仅在前端展示层做语义映射与编排。

## Decisions

### 决策 1：采用“中心舞台 + 渐进披露”的信息架构
- 方案：将 Presence Orb 与状态短句作为中心主舞台，实时理解区和回应区作为次级主内容，事件时间线默认折叠并按需展开。
- 原因：语音交互的核心是状态可感知性，而不是同时展示尽可能多的信息。中心舞台能建立稳定注意力锚点，减少扫视成本。
- 备选方案：继续沿用多卡片并列布局，仅微调配色与间距。
- 放弃理由：只能改善视觉整洁度，无法形成“被唤醒助手”的交互心智。

### 决策 2：新增体验状态机并与现有状态做非破坏映射
- 方案：在 UI 层定义体验状态机：
  - Dormant：待机可唤醒
  - WakeDetected：唤醒词命中后的短暂确认态
  - Listening：持续听取用户输入
  - Processing：转写/发送/工具处理中
  - Responding：语音播报或展示结果
  - Error：异常与恢复提示
- 映射策略：基于现有 `useBackendVAD`、`useAgentEvents`、连接状态与输出事件推导，不修改后端协议。
- 原因：体验状态与底层状态粒度不同，单纯暴露 VAD 状态不足以表达“激活感”。
- 备选方案：直接把 VAD 原始状态映射到文案。
- 放弃理由：缺少 WakeDetected 与 Responding 两个关键体感状态。

### 决策 3：定义“最小可信闭环”作为纯语音模式的必需反馈
- 方案：固定展示四类反馈：
  - Heard：最近一句短转写
  - Intent：系统理解的任务意图摘要
  - Status：当前执行阶段
  - Response：最近一次回答摘要
- 原因：纯语音体验中，信任建立依赖“可验证中间态”，否则用户难以判断是否误听或误执行。
- 备选方案：仅显示最终结果，不显示中间理解。
- 放弃理由：在嘈杂环境和复杂任务下，误解成本高且难以及时纠正。

### 决策 4：动效遵循“短仪式 + 可降级”原则
- 方案：
  - 激活仪式总时长控制在 600-1200ms。
  - 常规状态切换动效 150-300ms，避免拖沓。
  - 对 `prefers-reduced-motion` 自动降级为低动态版本（颜色与透明度变化）。
- 原因：需要在“可感知”与“不打扰”之间平衡，且满足可访问性需求。
- 备选方案：强动画或完全静态。
- 放弃理由：强动画影响长期使用舒适度；完全静态又损失激活感。

### 决策 5：保留兜底控制但降权
- 方案：`Hold to Talk`、`Mute`、`Stop` 等控件保留在低权重区域，不作为主视觉入口；配置项放入次级区或设置入口。
- 原因：用户依旧需要故障恢复与隐私控制，但不应与语音主流程竞争注意力。
- 备选方案：彻底移除按钮，强制全语音。
- 放弃理由：在权限失败、噪声环境或唤醒词失效时缺乏可操作恢复路径。

## Risks / Trade-offs

- [风险] 体验状态推导与真实后端阶段短暂不一致（例如 processing 与 responding 边界） → [缓解] 为每个状态定义明确优先级与超时回退策略，并补充状态迁移测试。
- [风险] 动效引入后在低性能设备上造成卡顿 → [缓解] 使用轻量 CSS 动效、减少布局抖动属性、提供 reduced-motion 分支。
- [风险] 语音优先导致高级信息（完整事件流）被隐藏过深 → [缓解] 提供自动展开触发条件（错误、confirm、失败工具调用）和显式“查看完整时间线”入口。
- [风险] 旧用户习惯并列卡片布局，短期学习成本上升 → [缓解] 保留关键状态文本与术语连续性，分阶段引导与文案过渡。

## Migration Plan

1. 先在前端引入体验状态映射层与新的布局骨架，保持现有数据源不变。
2. 分阶段迁移组件职责：先主舞台与理解闭环，再调整事件流默认展开策略。
3. 增补/更新组件测试：覆盖状态机迁移、关键文案可见性、confirm 自动展开、reduced-motion。
4. 通过 `pnpm build` 与 `pnpm test` 验证稳定性；必要时补充交互快照测试。
5. 若出现体验回归，可临时回退到旧布局入口（同一数据层，不影响后端）。

## Open Questions

- 唤醒词命中事件是否已经有稳定前端可见信号；若暂无，需要定义 UI 端近似触发策略（例如 listening->recording 边沿触发）。
- Responding 状态是否需要显式绑定 TTS 播放事件；若当前无 TTS，是否以首条 result 到达作为替代。
- 时间线默认折叠策略在高频开发场景下是否需要“记住用户偏好”。
